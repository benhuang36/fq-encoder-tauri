# Architecture — a Tauri walkthrough

**English** · [繁體中文](ARCHITECTURE.zh-Hant.md)

This project is small but exercises most of what you need to know to build a real
Tauri 2 app: commands, plugins, the permission model, a tray icon, background work,
shared state, and the JS↔Rust bridge. This doc maps the files to those concepts so
you can read the code with a map in hand.

> All symbols referenced here are real — open the named file and search for them.

---

## 1. The Tauri mental model

A Tauri app is **two halves talking over an IPC bridge**:

```
┌─────────────────────────┐         invoke()          ┌──────────────────────────┐
│   Web frontend           │  ───────────────────────► │   Rust core               │
│   (src/, index.html)     │                            │   (src-tauri/)            │
│   runs in the OS's        │  ◄─────────────────────── │   native binary; owns      │
│   native webview          │         events / return   │   window, tray, clipboard  │
└─────────────────────────┘                            └──────────────────────────┘
```

- **Rust core** owns the OS: the window, tray, clipboard, filesystem. It's a real
  native process.
- **Frontend** is HTML/CSS/JS rendered in the platform's *native* webview — WKWebView
  (macOS), WebView2 (Windows), WebKitGTK (Linux). Tauri does **not** bundle a browser,
  which is why the installers are a few MB instead of ~100 MB like Electron.
- The frontend calls Rust via **`invoke("command", args)`**; Rust can push **events**
  back. That's the whole bridge.

## 3. Key files → the concept each teaches

### `src-tauri/tauri.conf.json` — declarative app config

Defines the app shell without code:

- `app.windows[]` — the window: `"label": "main"`, size `480×660`, title. The label
  `"main"` is how Rust later finds the window (`get_webview_window("main")`) and how
  `capabilities` scopes permissions.
- `build` — ties the Rust build to the frontend: `beforeDevCommand: "npm run dev"`,
  `devUrl: "http://localhost:1420"`, `frontendDist: "../dist"`.
- `bundle` — installer icons/targets. `productName` + `version` name the output files.

### `src-tauri/capabilities/default.json` — the permission model

Tauri 2's biggest difference from Electron: **nothing is allowed by default.** You
explicitly grant each capability. Ours:

```json
"permissions": [
  "core:default", "opener:default", "store:default",
  "clipboard-manager:allow-read-text", "clipboard-manager:allow-write-text"
]
```

If a plugin call fails at runtime with a "not allowed" error, this file is the first
place to look — you forgot to grant the permission.

### `src-tauri/src/lib.rs` — the heart (read this slowly)

`run()` is one fluent `tauri::Builder` chain that demonstrates six concepts:

| Line of `run()` | Concept |
|---|---|
| `.plugin(tauri_plugin_clipboard_manager::init())` (and `store`, `opener`) | **Plugins** — opt-in OS capabilities |
| `.invoke_handler(generate_handler![encode, decode])` | **Commands** — the `#[tauri::command]` fns exposed to JS |
| `.manage(state.clone())` / `app.state::<Arc<MonitorState>>()` | **Shared state** — typed, retrievable anywhere |
| `.setup(\|app\| { … })` | **Startup hook** — runs once after init |
| `TrayIconBuilder::with_id("main-tray")…` | **Tray icon + menu** (`MenuItem`, `CheckMenuItem`) |
| `.on_window_event(\| CloseRequested \| …)` | **Window events** — hide instead of quit (resident app) |

Things to trace inside `setup()`:

- `app.set_activation_policy(ActivationPolicy::Accessory)` — macOS-only; removes the
  Dock icon so it's a pure menu-bar app.
- `tray_labels()` — localises the tray menu by OS locale (`sys-locale`).
- `spawn_clipboard_monitor(app.handle().clone(), state)` — launches the background
  thread (see flow #2 below).

### `src-tauri/src/codec.rs` — framework-agnostic logic

Note it has **zero Tauri imports**. Pure Rust (`encode`, `decode`, `looks_encoded`)
plus `#[cfg(test)]` tests you can run with plain `cargo test`. Good habit: keep your
domain logic decoupled from the framework so it's testable and portable (this exact
file mirrors the macOS Swift app, verified by golden-vector tests).

### `src/main.ts` — the frontend side of the bridge

- `invoke<string>("encode", { text, key })` — calls the Rust command; the JS arg keys
  (`text`, `key`) must match the Rust fn parameter names.
- `load(STORE_FILE, …)` + `store.get/set` — the `@tauri-apps/plugin-store` JS API,
  reading/writing the same `settings.json` the Rust side reads.

## 4. Two data flows worth tracing

**① Manual encode (button → Rust → back):**

```
click handler (main.ts)
  → invoke("encode", { text, key })            [JS]
  → #[tauri::command] fn encode(text, key)     [Rust, lib.rs:26]
  → codec::encode(&text, &key)                 [Rust, codec.rs:58]
  → returns String → resolves the JS promise → setOutput()
```

`decode` is the same but returns `Result<String, String>` — on error it returns a
**stable code** (`e.code()`, e.g. `invalid_character:x`) that `main.ts` localises.
That's the pattern for keeping UI language out of Rust.

**② Clipboard auto-monitor (no UI involved):**

```
std::thread loop every 500 ms          (spawn_clipboard_monitor, lib.rs:66)
  → app.clipboard().read_text()         [clipboard plugin]
  → loop-safe guards (MonitorState.last_seen / last_written)
  → codec::looks_encoded() ? decode : encode
  → app.clipboard().write_text(result)
```

**The shared-store sync:** both halves need the password. The frontend writes it to
the store (`fq.encodingPassword`); the Rust monitor reads it each poll via
`read_password()` (lib.rs:50). Using the store as shared config avoids extra IPC.

## 5. Concept index

- **Command** — a Rust fn marked `#[tauri::command]` and registered in
  `invoke_handler`; callable from JS via `invoke`.
- **Plugin** — a packaged OS capability (clipboard, store, opener) registered with
  `.plugin(...)` and matched by a permission in `capabilities`.
- **Capability / permission** — the allow-list in `capabilities/default.json`;
  Tauri 2 denies everything not listed.
- **Tray** — a status-bar/system-tray icon built with `TrayIconBuilder` + a `Menu`.
- **State** — typed shared data via `.manage(T)` and `app.state::<T>()`.
- **Event** — Rust→JS (or JS→Rust) message via `emit`/`listen` (this app uses
  commands, not custom events — but it's the other half of the bridge).
- **Store** — the `plugin-store` key/value JSON file, usable from both JS and Rust.

## 6. Suggested reading order

1. `src-tauri/tauri.conf.json` — the app shell.
2. `src-tauri/src/lib.rs` — `run()`: Builder, commands, plugins, setup, tray.
3. `src-tauri/capabilities/default.json` — why those plugin calls are allowed.
4. `src/main.ts` — the `invoke` + store JS side.
5. `src-tauri/src/codec.rs` — see how logic stays decoupled.
